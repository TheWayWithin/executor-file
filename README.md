# Digital Estate Register

An encrypted, self-hosted file that lets your executor find every account, asset, and liability you own — and know exactly what you want done with each one — with **no credentials stored** and **no dependence on any service staying alive**.

It is a file, not an app. There is nothing to subscribe to, nothing to keep patched, and nothing here that can leak a password, because no password ever enters it.

## The promises this design keeps

A security-literate reader should be able to verify each of these directly from this repo:

1. **No secrets at rest.** The register stores *pointers* ("1Password > HSBC", "seed backup in deposit box …22") — never credentials, full account numbers, or seed phrases. The validator ([`scripts/validate.py`](scripts/validate.py)) actively rejects long digit runs and flags written-out credentials.
2. **No service dependency.** Encryption is [`age`](https://age-encryption.org) in passphrase mode; secret-splitting is [`ssss`](http://point-at-infinity.org/ssss/) (Shamir's Secret Sharing). Both are open-source, packaged everywhere, and operate on local files. If this repo and its author vanish, the encrypted file still opens with two standard commands.
3. **No single point of compromise.** The passphrase is split 2-of-3: three people each hold one printed share. Any two reconstruct it; any one alone learns nothing (information-theoretically, not just computationally). While you are alive, no individual can open the record.
4. **No single point of failure.** One lost share, one dead shareholder, one lost backup of the `.age` file — none of them is fatal. The encrypted file is useless without two shares, so it can be backed up promiscuously.
5. **Nothing private is committed.** [`.gitignore`](.gitignore) excludes `estate.yaml` and `*.age`. Only the schema, an example with dummy data, templates, and scripts belong in git.

The scripts are thin, readable wrappers — around 60 lines each. Read them before trusting them; that is the point of them being short.

## How it works

```
estate.yaml   --age -p-->   estate.yaml.age          (the record, encrypted)
passphrase    --ssss-split-->  share1 / share2 / share3   (2-of-3)
```

Three artefacts, nothing more:

| Artefact | What it is | Where it lives |
|---|---|---|
| `estate.yaml` | Plaintext source record | Never committed, never stored long-term |
| `estate.yaml.age` | The encrypted record | The only version that persists — back it up freely |
| `EXECUTOR-INSTRUCTIONS.md` | Printed page telling the executor how to get in | On paper, with the will |

On death, the executor reads the printed page, collects shares from two of the three holders, and runs three commands: install, `ssss-combine -t 2`, `age -d`. No account, no company, no login. The full walkthrough — written for a non-technical person under stress — is [`templates/EXECUTOR-INSTRUCTIONS.md`](templates/EXECUTOR-INSTRUCTIONS.md).

## Quickstart (you, the owner)

Requirements: `age` and `ssss` (`brew install age ssss` on macOS, `sudo apt install age ssss` on Debian/Ubuntu), plus `python3` with PyYAML for the optional validator.

```bash
# 1. Start from the example and fill in your real assets
cp examples/estate.example.yaml estate.yaml
$EDITOR estate.yaml            # estate.yaml is git-ignored

# 2. Check it — structure, allowed values, and the no-secrets rules
scripts/validate.sh

# 3. Encrypt (press Enter at the prompt to autogenerate a strong passphrase —
#    you'll never memorise it, the shares carry it from here)
scripts/encrypt.sh

# 4. Split the passphrase into 3 shares, any 2 of which reconstruct it
scripts/split-secret.sh

# 5. Prove you can get back in BEFORE deleting anything
scripts/decrypt.sh estate.yaml.age | head

# 6. Fill in templates/EXECUTOR-INSTRUCTIONS.md, print it, store with the will
```

Then the physical part, which matters more than the software: put one printed share with each of three holders (e.g. executor, solicitor, family member) and tell them what it is; store `estate.yaml.age` in at least two places; delete the plaintext `estate.yaml`; and set the platform legacy tools below.

Keep the passphrase within 128 ASCII characters — `ssss` will not split more (age's autogenerated passphrases fit comfortably).

## What goes in the register

The schema is documented (and machine-enforced) in [`schema/estate.schema.yaml`](schema/estate.schema.yaml); a realistic dummy register is in [`examples/estate.example.yaml`](examples/estate.example.yaml). The two load-bearing fields per asset:

- **`type`** — `cash | liability | subscription | holding | crypto | online-business | other`. Crypto is treated as high priority everywhere: a missed wallet is unrecoverable, not merely delayed.
- **`action`** — `liquidate | cancel | transfer | delete | notify-only`. This is the instruction your executor acts on; `action_notes` says it in your own words.

`identifier` is a last-4 or reference only. `access_pointer` says where the login *lives* (your password manager), never what it is.

## Platform legacy tools come first

Under **RUFADAA** (the Revised Uniform Fiduciary Access to Digital Assets Act, enacted in nearly every US state), access to digital accounts follows a strict hierarchy: **(1) the platform's own online designation tool outranks (2) your will and estate documents, which outrank (3) the provider's terms of service.** A forgotten legacy-contact setting beats your will — so set these deliberately, keep them consistent with the will, and record them in the register:

| Platform | Tool | Where to set it |
|---|---|---|
| Apple | Legacy Contact | Settings > [your name] > Sign-In & Security > Legacy Contact. The contact later needs their **access key + your death certificate** at [digital-legacy.apple.com](https://digital-legacy.apple.com). Keychain passwords and licensed media are not included. |
| Google | Inactive Account Manager | [myaccount.google.com/inactive](https://myaccount.google.com/inactive). Timer-based (3–18 months of inactivity), up to 10 contacts, per-contact data selection. Also prevents Google's 2-year inactive-account deletion from surprising your estate. |
| Meta / Facebook | Legacy Contact | Settings > Memorialization settings — name a legacy contact (manages a memorialised profile; cannot read messages) or choose deletion on death. Instagram offers memorialisation only. |

In the UK there is no RUFADAA equivalent: executors have no statutory right of access, and each provider's terms plus data-protection law govern — which makes the platform tools and this register more important, not less.

**Example estate-document wording** (not legal advice — take it to whoever drafts the will):

> "I authorise my executor to access, manage, control, transfer, and dispose of my digital assets and digital accounts, and I expressly consent to the disclosure to my executor of the content of my electronic communications, to the fullest extent permitted by the Revised Uniform Fiduciary Access to Digital Assets Act or any similar applicable law."

## Threat model

| Threat | Mitigation |
|---|---|
| Someone opens the record while you're alive | 2-of-3 Shamir split; no one holds the whole passphrase |
| One shareholder lost / unreachable / dies first | 2-of-3 tolerates one loss |
| The encrypted file leaks | `age` scrypt passphrase encryption; the file alone is useless without two shares |
| Credentials exposed via this file | None stored; the register only points at your password manager |
| A service shuts down and breaks access | There is no service — static binaries and a local file |
| Wrongful or early access | Shares are physical and held by trusted parties; access requires two humans agreeing |
| The record goes stale | Not solved by software — see below |

**The honest residual risk is staleness.** A register that misses your newest accounts quietly lies to your executor. Re-run steps 1–5 whenever the estate changes materially, and at minimum at every will review — put it in the calendar now. The `meta.updated` field exists so your executor can see how much to trust what they're reading.

## Repository layout

```
├── README.md                       what you're reading
├── LICENSE                         MIT
├── schema/estate.schema.yaml       documented schema — also drives the validator
├── examples/estate.example.yaml    dummy register, safe to commit
├── templates/EXECUTOR-INSTRUCTIONS.md  printable page, blanks to fill
├── scripts/
│   ├── encrypt.sh                  age -p wrapper (validates first)
│   ├── decrypt.sh                  age -d wrapper (stdout by default)
│   ├── split-secret.sh             ssss-split wrapper (2-of-3 default)
│   ├── validate.sh                 lint a register against the schema
│   └── validate.py                 the validator (needs PyYAML)
└── .gitignore                      excludes estate.yaml and *.age
```

Your executor never needs this repo — the printed instructions use `age` and `ssss` directly.

## Deliberately out of scope

No credential storage (that's your password manager's job). No provider APIs, generated emails, or liquidation automation (that's where legal liability and per-jurisdiction rules live). No hosted anything. The value of this tool is *discovery plus disposition*; everything else is somebody's bereavement process.
